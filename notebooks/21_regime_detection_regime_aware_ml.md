# Step 9 - Regime Detection and Regime-Aware ML

## Goal

Build adaptive machine learning systems that detect the current market regime and switch to models
trained for that regime.

## Why One Model Fails In All Conditions

Markets do not behave the same way all the time. A momentum model may work during persistent trends
and fail during sideways mean reversion. A volatility breakout signal may work in crisis regimes
and overtrade in calm markets. A single global model averages these behaviors together and can learn
weak compromises.

Regime-aware systems ask:

```text
What type of market are we in, and which model should be trusted here?
```

## Regimes Implemented

- `trending`: directional movement is strong relative to volatility.
- `ranging`: price is choppy or mean-reverting.
- `high_volatility`: realized volatility is elevated.

## Implemented Components

- `create_regime_feature_matrix`: rolling return, rolling volatility, trend strength.
- `classify_market_regimes`: deterministic volatility/trend regime labels.
- `ClusteringRegimeDetector`: KMeans-based unsupervised regimes.
- `HMMRegimeDetector`: Gaussian Hidden Markov Model regimes.
- `RegimeAwareModel`: trains separate predictive models per regime and switches dynamically.

## HMM Intuition

A Hidden Markov Model assumes the observed market features are generated by hidden states. Those
states are not directly observed, but the model infers them from returns, volatility, and trend
strength.

In finance, HMMs are useful because regimes tend to persist. A high-volatility regime today
increases the probability of high volatility tomorrow.

## Clustering Intuition

Clustering groups similar market observations together. If one cluster has high realized
volatility, we label it `high_volatility`. If another has high trend strength, we label it
`trending`. Remaining clusters are usually `ranging`.

## Regime-Aware Modeling

The regime-aware model trains:

```text
fallback model on all data
trending model on trending rows
ranging model on ranging rows
high-volatility model on high-volatility rows
```

At prediction time, it uses the regime label at each timestamp to route the row to the appropriate
model.

## Minimal Example

```python
from regime import (
    RegimeAwareModel,
    RegimeDetectionConfig,
    classify_market_regimes,
    create_regime_feature_matrix,
)

regimes = classify_market_regimes(prices, RegimeDetectionConfig())

model = RegimeAwareModel()
model.fit(features, target, regimes)

predictions = model.predict(features, regimes).predictions
```

## Interview-Ready Explanation

I detect market regimes using rolling return, volatility, and trend-strength diagnostics. I support
both deterministic labeling and unsupervised models such as KMeans and Gaussian HMMs. Then I train
separate predictive models per regime and dynamically switch between them at prediction time. This
helps avoid forcing one model to handle trending, ranging, and stressed markets with the same
parameters.

## Practical Warning

Regime models can overfit. If each regime has too few observations, the specialized models may be
less stable than a single global model. That is why the implementation includes a fallback model
trained on all data and a minimum-observation threshold for regime-specific models.
